---
title: Deep Merging
description: How different file extensions are deeply merged together
---

When the orchestrator successfully intercepts a file (either via [Templates](../templates) or [Config Injection](../config)), and that file already exists in the runtime container, it intelligently deep-merges them based on the extension!

Each payload also supports [Variable Interpolation](../interpolation) using the `$[VARIABLE_NAME]` syntax. This occurs before any merging logic hits the file!

It actively supports preserving comments across most formats (YAML, TOML, and Properties)! You can also use key-level [sigils](../sigils) (`!replace:` and `!delete:`) directly within these structured configuration files to bypass standard merging logic on specific dictionaries.

### YAML (`.yml`, `.yaml`)

Because the orchestrator relies on `ruamel.yaml`, your structural comments remain entirely untouched. 

In this example, we change the port via standard merging, `!delete:` the legacy rules completely, and `!replace:` the entire advanced dictionary (wiping out old properties that aren't explicitly inside the new template block).

<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
<div>
**Runtime (Before)**
```yaml
# server config
server:
  port: 25565
  # Should we enable online mode?
  online-mode: true
  legacy_rules: true
  advanced:
    cache: true
    debug: false
```
</div>
<div>
**Template**
```yaml
server:
  port: 25566
  "!delete:legacy_rules": ""
  "!replace:advanced":
    cache: false
```
</div>
</div>

**Result (in Runtime)**
```yaml
# server config
server:
  port: 25565 # [!code --]
  port: 25566 # [!code ++]
  # Should we enable online mode?
  online-mode: true
  legacy_rules: true # [!code --]
  advanced: # [!code --]
    cache: true # [!code --]
    debug: false # [!code --]
  advanced: # [!code ++]
    cache: false # [!code ++]
```

### JSON (`.json`)

JSON supports the exact same nested behavior. Here we `!replace` the features dict completely and `!delete` a deprecated key, while seamlessly deep merging `new_system`.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
<div>
**Runtime (Before)**
```json
{
  "max_players": 20,
  "features": {
    "flight": false,
    "rcon": false
  },
  "old_system": true,
  "new_system": false
}
```
</div>
<div>
**Template Payload**
```json
{
  "!replace:features": {
    "rcon": true
  },
  "!delete:old_system": null,
  "new_system": true
}
```
</div>
</div>

**Result (in Runtime)**
```json
{
  "max_players": 20,
  "features": {
    "flight": false, // [!code --]
    "rcon": false,   // [!code --]
    "rcon": true     // [!code ++]
  },
  "old_system": true,  // [!code --]
  "new_system": false, // [!code --]
  "new_system": true   // [!code ++]
}
```

### TOML (`.toml`)

TOML supports sigils on both flat keys and table structures. Below we change the debug flag naturally, `!delete:` the legacy flag, and `!replace:` the entire database table layout.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
<div>
**Runtime (Before)**
```toml
# Main settings
[core]
debug = false # Change to true to log everything
legacy = true

[database]
uri = "localhost:5432"
timeout = 30
```
</div>
<div>
**Template Payload**
```toml
[core]
debug = true
"!delete:legacy" = true

["!replace:database"]
uri = "remote:5432"
```
</div>
</div>

**Result (in Runtime)**
```toml
# Main settings
[core]
debug = false # Change to true to log everything # [!code --]
debug = true  # Change to true to log everything # [!code ++]
legacy = true # [!code --]

[database] # [!code --]
uri = "localhost:5432" # [!code --]
timeout = 30 # [!code --]

[database] # [!code ++]
uri = "remote:5432" # [!code ++]
```

### Java Properties (`.properties`)

Because Java Properties are essentially flat lists without objects or arrays, `!replace:` is completely unnecessary - overriding properties defaults to 1:1 replacement anyway! However, `!delete:` remains invaluable to remove stale flags perfectly.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
<div>
**Runtime (Before)**
```properties
# Minecraft server properties
view-distance=10
enable-command-block=false
legacy-mode=true
motd=A Minecraft Server
```
</div>
<div>
**Template Payload**
```properties
view-distance=12
!delete:legacy-mode=true
motd=mc-orchestrator rules!
```
</div>
</div>

**Result (in Runtime)**
```properties
# Minecraft server properties
view-distance=10            # [!code --]
view-distance=12            # [!code ++]
enable-command-block=false
legacy-mode=true            # [!code --]
motd=A Minecraft Server     # [!code --]
motd=mc-orchestrator rules! # [!code ++]
```

### HOCON (`.conf`)

HOCON (Human-Optimized Config Object Notation) is a super-set of JSON often found in plugins like LuckPerms or Sponge. The orchestrator allows deep-merging `.conf` files exactly like JSON.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4">
<div>
**Runtime (Before)**
```js
storage-method = "h2"
data {
  address = "127.0.0.1"
  pool-settings {
    maximum-pool-size = 10
  }
}
```
</div>
<div>
**Template Payload**
```js
storage-method = "postgresql"
data {
  address = "192.168.1.100"
  "!replace:pool-settings" {
    max = 20
  }
}
```
</div>
</div>

**Result (in Runtime)**
```js
storage-method = "h2" // [!code --]
storage-method = "postgresql" // [!code ++]
data {
  address = "127.0.0.1" // [!code --]
  address = "192.168.1.100" // [!code ++]
  pool-settings {
    maximum-pool-size = 10 // [!code --]
    max = 20 // [!code ++]
  }
}
```

<Callout title="Note" type="warning">
  Unlike YAML and TOML, the HOCON merger **does not** preserve comments in the existing file. Merging into a `.conf` file will result in a clean HOCON output without the original comments or custom formatting.
</Callout>
